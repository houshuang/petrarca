# Content Refresh Architecture

How to get new articles into the app without redeploying.

## Current Flow

```
[Local Mac]
  ~/src/otak/data/twitter_bookmarks.json  (fetched by otak scripts)
  ~/src/otak/data/readwise_reader.json    (fetched by otak scripts)
       |
       v
  scripts/build_articles.py
       |  - collects candidates from Twitter bookmarks + Readwise
       |  - fetches article HTML, extracts text (trafilatura)
       |  - LLM processing via `claude -p` (sections, summaries, claims)
       |  - concept extraction across articles
       |
       v
  data/articles.json          (pipeline working copy)
  data/concepts.json          (pipeline working copy)
  app/data/articles.json      (bundled into Expo app)
  app/data/concepts.json      (bundled into Expo app)
       |
       v
  git commit + push -> deploy.sh -> Hetzner pulls -> systemctl restart petrarca-expo
```

The app loads articles via `require('./articles.json')` in `store.ts:26-27`. This is a static import resolved at bundle time. New articles require a full redeploy.

## Design: Scheduled Pipeline + HTTP Content Server

### 1. Pipeline on Hetzner (cron)

The pipeline needs these data sources on the server:
- Twitter bookmarks: `~/src/otak/data/twitter_bookmarks.json`
- Readwise Reader: `~/src/otak/data/readwise_reader.json`
- Claude CLI (`claude -p`) for LLM processing

**Source data sync**: The otak bookmark fetcher runs on the local Mac (requires browser cookies). Simplest approach: rsync the source data to Hetzner before the pipeline runs, or have otak's fetch scripts also push to the server.

**Cron setup** on Hetzner (`/etc/cron.d/petrarca-refresh`):

```cron
# Sync source data from Mac (requires SSH key)
# This assumes the Mac is reachable — alternative: push from Mac on a schedule
# 0 6 * * * stian rsync -az stian@mac:~/src/otak/data/ /opt/petrarca/data/sources/

# Run pipeline daily at 7 AM UTC (incremental mode, max 10 new articles)
0 7 * * * stian cd /opt/petrarca && python3 scripts/build_articles.py --limit 10 >> /var/log/petrarca-refresh.log 2>&1
```

**Path changes in `build_articles.py`**: The pipeline currently hardcodes `~/src/otak/data/` as source paths. For server mode, add a `--sources-dir` flag or detect based on hostname:

```python
# In build_articles.py, change OTAK_BOOKMARKS and OTAK_READWISE:
SOURCES_DIR = Path(os.environ.get("PETRARCA_SOURCES", str(Path.home() / "src" / "otak" / "data")))
OTAK_BOOKMARKS = SOURCES_DIR / "twitter_bookmarks.json"
OTAK_READWISE = SOURCES_DIR / "readwise_reader.json"
```

Then set `PETRARCA_SOURCES=/opt/petrarca/data/sources` in the cron environment.

**Pipeline output**: The pipeline already saves to `data/articles.json` and `data/concepts.json`. These become the served files.

### 2. Serve Content via HTTP (nginx or static file server)

Simplest option: nginx serves the output JSON files directly. The Hetzner VM already runs nginx for the Expo dev server.

**nginx config** (add to existing server block or new one):

```nginx
# /etc/nginx/sites-available/petrarca-content
server {
    listen 8083;
    server_name alifstian.duckdns.org;

    location /content/ {
        alias /opt/petrarca/data/;
        add_header Access-Control-Allow-Origin "*";
        add_header Cache-Control "public, max-age=300";
        types { application/json json; }
    }

    # Manifest endpoint: article count + last_updated timestamp
    # (generated by pipeline at end of run)
    location /content/manifest.json {
        alias /opt/petrarca/data/manifest.json;
        add_header Access-Control-Allow-Origin "*";
        add_header Cache-Control "no-cache";
    }
}
```

**URLs**:
- `http://alifstian.duckdns.org:8083/content/articles.json`
- `http://alifstian.duckdns.org:8083/content/concepts.json`
- `http://alifstian.duckdns.org:8083/content/manifest.json`

**Manifest file** — generated by the pipeline after each run:

```json
{
  "last_updated": "2026-03-03T07:00:00Z",
  "article_count": 61,
  "concept_count": 75,
  "articles_hash": "sha256:abc123...",
  "concepts_hash": "sha256:def456..."
}
```

Add to end of `build_articles.py`'s `main()`:

```python
import hashlib

manifest = {
    "last_updated": datetime.now(timezone.utc).isoformat(),
    "article_count": len(articles),
    "concept_count": len(concepts),
    "articles_hash": hashlib.sha256(json.dumps(articles, sort_keys=True).encode()).hexdigest()[:16],
    "concepts_hash": hashlib.sha256(json.dumps(concepts, sort_keys=True).encode()).hexdigest()[:16],
}
_save_json(manifest, DATA_DIR / "manifest.json")
```

### 3. App Changes: Fetch Remote Content

**Strategy**: Keep static JSON as the **fallback** (bundled articles still load instantly on first install). On app launch, check for newer content from the server. If available, download and merge.

**New file: `app/data/content-sync.ts`**

```typescript
import * as FileSystem from 'expo-file-system';
import { Article, Concept } from './types';

const CONTENT_BASE = 'http://alifstian.duckdns.org:8083/content';
const MANIFEST_URL = `${CONTENT_BASE}/manifest.json`;
const ARTICLES_URL = `${CONTENT_BASE}/articles.json`;
const CONCEPTS_URL = `${CONTENT_BASE}/concepts.json`;

const CACHE_DIR = `${FileSystem.documentDirectory}content/`;
const CACHED_ARTICLES = `${CACHE_DIR}articles.json`;
const CACHED_CONCEPTS = `${CACHE_DIR}concepts.json`;
const CACHED_MANIFEST = `${CACHE_DIR}manifest.json`;

interface Manifest {
  last_updated: string;
  article_count: number;
  concept_count: number;
  articles_hash: string;
  concepts_hash: string;
}

export async function checkForUpdates(): Promise<boolean> {
  try {
    const resp = await fetch(MANIFEST_URL, { cache: 'no-store' });
    if (!resp.ok) return false;
    const remote: Manifest = await resp.json();

    // Compare with cached manifest
    try {
      const localRaw = await FileSystem.readAsStringAsync(CACHED_MANIFEST);
      const local: Manifest = JSON.parse(localRaw);
      return remote.articles_hash !== local.articles_hash
          || remote.concepts_hash !== local.concepts_hash;
    } catch {
      // No cached manifest = first sync
      return true;
    }
  } catch {
    return false; // offline or server unreachable
  }
}

export async function downloadContent(): Promise<{ articles: Article[]; concepts: Concept[] } | null> {
  try {
    await FileSystem.makeDirectoryAsync(CACHE_DIR, { intermediates: true });

    const [articlesResp, conceptsResp, manifestResp] = await Promise.all([
      fetch(ARTICLES_URL),
      fetch(CONCEPTS_URL),
      fetch(MANIFEST_URL),
    ]);

    if (!articlesResp.ok || !conceptsResp.ok) return null;

    const articles: Article[] = await articlesResp.json();
    const concepts: Concept[] = await conceptsResp.json();
    const manifest = await manifestResp.json();

    // Write to local cache
    await Promise.all([
      FileSystem.writeAsStringAsync(CACHED_ARTICLES, JSON.stringify(articles)),
      FileSystem.writeAsStringAsync(CACHED_CONCEPTS, JSON.stringify(concepts)),
      FileSystem.writeAsStringAsync(CACHED_MANIFEST, JSON.stringify(manifest)),
    ]);

    return { articles, concepts };
  } catch {
    return null;
  }
}

export async function loadCachedContent(): Promise<{ articles: Article[]; concepts: Concept[] } | null> {
  try {
    const [articlesRaw, conceptsRaw] = await Promise.all([
      FileSystem.readAsStringAsync(CACHED_ARTICLES),
      FileSystem.readAsStringAsync(CACHED_CONCEPTS),
    ]);
    return {
      articles: JSON.parse(articlesRaw),
      concepts: JSON.parse(conceptsRaw),
    };
  } catch {
    return null;
  }
}
```

**Changes to `app/data/store.ts` `initStore()`**:

```typescript
// In initStore(), after loading bundled articles:

// Try cached remote content first (newer than bundle)
const cached = await loadCachedContent();
if (cached && cached.articles.length > 0) {
  articles = cached.articles;
  concepts = cached.concepts;
}

// ... rest of init (build index, load user state) ...

// Background: check for updates after init completes
checkForUpdates().then(async (hasUpdates) => {
  if (hasUpdates) {
    const fresh = await downloadContent();
    if (fresh) {
      // Merge: keep all articles (new + existing), rebuild index
      mergeNewContent(fresh.articles, fresh.concepts);
      logEvent('content_refreshed', {
        article_count: fresh.articles.length,
        concept_count: fresh.concepts.length,
      });
    }
  }
});
```

### 4. Handling New Concepts from New Articles

New pipeline runs will produce new concepts. The concept extraction already deduplicates across articles (word-overlap matching in `_deduplicate_concepts`).

**Key guarantee**: Article IDs are deterministic (`sha256(source_url)[:12]`). Concept IDs are sequential (`c0001`, `c0002`, ...) and re-generated each run across all articles.

**Problem**: Concept IDs are not stable across runs. If the pipeline re-processes all articles, concept IDs may shift. User's concept states (stored by concept ID in AsyncStorage) would become orphaned.

**Solution**: Make concept IDs deterministic based on content, not sequence. Change `build_articles.py`:

```python
# In extract_concepts(), replace sequential ID:
concept["id"] = hashlib.sha256(item["text"].encode()).hexdigest()[:10]
```

This makes concept IDs stable: same concept text always gets the same ID, regardless of batch ordering or when the pipeline runs. The existing `_deduplicate_concepts` function already merges similar concepts by text overlap, so the same concept appearing in different runs will get the same ID.

### 5. Merging Content Without Losing User State

User state is keyed by article ID and concept ID in AsyncStorage. The merge strategy:

**Articles**: Article IDs are already deterministic (hash of source_url). The app stores reading states, signals, and voice notes keyed by article ID. New articles simply appear in the feed. Existing articles keep their state.

```typescript
// In store.ts, new function:
function mergeNewContent(newArticles: Article[], newConcepts: Concept[]) {
  // Articles: replace entire list (IDs are deterministic, so existing articles match)
  articles = newArticles;
  articles.sort((a, b) => (b.date || '').localeCompare(a.date || ''));

  // Concepts: replace list, user states persist by concept ID
  concepts = newConcepts;

  // Rebuild article->concept index
  articleConceptIndex.clear();
  for (const c of concepts) {
    for (const aid of c.source_article_ids) {
      if (!articleConceptIndex.has(aid)) articleConceptIndex.set(aid, []);
      articleConceptIndex.get(aid)!.push(c.id);
    }
  }

  // User state maps (readingStates, conceptStates, conceptReviews, signals, voiceNotes)
  // are untouched — they persist in AsyncStorage keyed by article_id / concept_id.
  // Orphaned entries (articles/concepts removed from pipeline) are harmless — they just
  // won't match any article/concept and are effectively invisible.
}
```

**What's preserved**:
- Reading states (by article_id) -- all survive, new articles start as 'unread'
- Signals (by article_id) -- all survive
- Concept states (by concept_id) -- survive if concept IDs are deterministic
- Concept reviews (by concept_id) -- survive if concept IDs are deterministic
- Voice notes (by article_id) -- all survive

**What could break**:
- If an article's source_url changes (re-scraped with different URL normalization), its ID changes and user state is lost for that article. Unlikely but possible.
- If concept text changes slightly between runs, concept ID changes. This is the motivation for content-based hashing above.

## Source Data: Getting Fresh Bookmarks to Hetzner

The Twitter bookmark fetcher (`otak/scripts/fetch_twitter_bookmarks.py`) requires browser cookies that expire periodically. Two approaches:

**Option A: Push from Mac (recommended for now)**
Add a local cron/launchd job on the Mac that syncs source data to Hetzner:

```bash
# ~/bin/sync-petrarca-sources.sh
#!/bin/bash
rsync -az ~/src/otak/data/twitter_bookmarks.json \
          ~/src/otak/data/readwise_reader.json \
          alif:/opt/petrarca/data/sources/
```

Run this daily via `launchd` or after each otak fetch.

**Option B: Readwise-only (simpler, no cookie issues)**
Readwise has an API. The pipeline could fetch directly from Readwise API on the server, bypassing the need for the otak data sync entirely. Twitter bookmarks still need the cookie-based approach, but Readwise is the larger source (40+ items per run vs ~10 from Twitter).

## Implementation Order

1. **Make concept IDs deterministic** in `build_articles.py` (~5 min)
2. **Add manifest generation** to pipeline (~5 min)
3. **Add `--sources-dir` flag** to pipeline (~5 min)
4. **Set up nginx** on Hetzner to serve `/content/` (~10 min)
5. **Create `content-sync.ts`** in app (~30 min)
6. **Modify `store.ts` `initStore()`** to use cached/remote content (~20 min)
7. **Set up cron** on Hetzner (~5 min)
8. **Set up source data sync** from Mac (~10 min)

Total: ~90 minutes of implementation work. Steps 1-3 are pipeline-only (no app changes, can test locally). Steps 4-7 are the actual "go live" changes.

## Future: Incremental Updates

The design above downloads the full `articles.json` on every sync. At 51 articles this is ~500KB, fine for now. If it grows past ~500 articles:

- Pipeline outputs `articles-{date}.json` with only new articles since last run
- App fetches manifest, checks which incremental files it's missing, downloads only those
- Or: pipeline outputs a single `articles.json` but the app uses `If-Modified-Since` / ETag headers to avoid re-downloading unchanged content (nginx handles this automatically with `etag on`)

Not needed until the article count grows significantly.
